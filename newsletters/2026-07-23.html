🧠 Kilo Memory: your agents finally remember the project
Kilo Memory is a local, project-scoped memory system that shares context across sessions and worktrees. No more starting from zero every time a session times out.
What it stores. Session digests, project environment details, user corrections, and anything you explicitly tell it to remember with /memory commands. A consolidation step filters out noise so only durable, useful context gets kept.
How it recalls. High-signal context gets injected at session start, pulled in when your prompt looks memory-relevant, or retrieved explicitly through the kilo_memory_recall tool. The memory store lives in Kilo's global data directory — no MCP server, no external service, no cloud sync.

🔄 We taught Kilo to test itself
The newest models are finally good enough to drive the VS Code extension and CLI end-to-end. So we gave Kilo the ability to run, observe, and validate its own work.
The agent can open VS Code windows, compile and run the CLI through the terminal, execute full end-to-end passes on a feature it just built, and profile itself to compare before and after a change. That means you can hand it a loop like "baseline this, optimize it, re-benchmark, repeat until it's under 50 milliseconds" — a workflow that didn't exist a year ago.

🏊 Laguna S 2.1 is live (and free for a limited time)
Poolside's 118B-parameter model is now available across the IDE extension, CLI, and Cloud Agents. It activates only 8B parameters per token, but it's already matching models many times its size on SWE-bench Multilingual and Toolathlon Verified.
The standout behavior is persistence: Poolside reports Laguna S 2.1 holding onto tasks, using tools, and recovering from failed approaches for up to 24 hours. That's exactly the kind of long-horizon work that shines in KiloClaw background sessions and Cloud Agents — and the efficient active parameter count keeps those long runs cheap.

🎯 Match skills to models with custom agents
Build one agent per model and switch model, prompt, and skill policy in one keystroke. It's the cleanest way to stop a frontier model from wasting context on skills it doesn't need.
There's no per-model skill toggle yet, but custom agents get you the same result. Pin GLM 5.2 to an agent that leans on superpowers for multi-step tasks, and pin Fable 5 to an agent that skips them entirely with a permission deny. Switch between them from the agent picker in VS Code or Tab in the CLI. Same files work everywhere because the CLI reads the same .kilo/agent/ config.
